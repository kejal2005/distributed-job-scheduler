1. Executive Summary

This project is a production-grade distributed job scheduling platform that reliably processes asynchronous background jobs across multiple worker nodes. The system is modeled after real-world platforms like Sidekiq, BullMQ, and GitHub Actions — it provides multi-tenant project isolation, atomic job claiming with PostgreSQL row-level locks, configurable retry strategies with dead-letter support, cron-based recurring jobs, and a live operations dashboard powered by WebSockets.

The architecture is split into three independently deployable components:

  • A Control Plane (REST API) that handles authentication, job ingestion, queue management, and worker coordination
  • A Worker Service that polls the Control Plane, claims jobs atomically, and executes them concurrently
  • A Frontend Dashboard that provides real-time observability into the entire system

All three are deployed to production and verified end-to-end from a mobile device over cellular data — confirming zero localhost dependency.

Key Design Choice
Workers communicate via REST, not direct database access. They authenticate with a project-scoped API key and interact solely through the Control Plane's HTTP endpoints. This means workers can run on entirely separate networks — the database is never exposed publicly, and all authorization logic is enforced centrally.

Normalization & Cascading

The schema follows 3NF with one intentional denormalization: jobs.attempt_count mirrors what could be derived from COUNT(*) FROM job_executions. This avoids a join in the hot-path claim query where every millisecond of lock time matters.

Foreign keys use deliberate cascade rules: ON DELETE CASCADE for ownership chains (org → project → queue → job), and ON DELETE SET NULL for soft references (e.g., claimed_by_worker_id — deleting a worker should not destroy its jobs).

PostgreSQL-Specific Features

  • pgcrypto for cryptographically random UUIDs
  • 7 custom ENUM types (job_status, job_type, worker_status, org_role, retry_strategy, execution_result, log_level)
  • JSONB columns for flexible payloads and worker metadata
  • TEXT[] array for worker queue assignments
  • FOR UPDATE SKIP LOCKED for lock-free concurrent claiming
  • ON CONFLICT DO NOTHING for idempotent job creation
  • PL/pgSQL trigger function touch_updated_at() on 3 tables

Request Validation (Zod)

Every mutating endpoint uses Zod schemas applied as Express middleware. The middleware parses the request body, returns structured 400 errors with field-level details on failure, and attaches validated data as req.validated. Schemas enforce email format, minimum password length (8 chars), UUID format for all ID references, enum constraints for job types and retry strategies, and positive integer constraints for concurrency limits.

Security

  • Password hashing: bcryptjs with default salt rounds
  • JWT tokens: 7-day expiry, user ID in sub claim
  • API key authentication: Workers authenticate with project-scoped keys via X-API-Key header
  • Helmet: Security headers (X-Frame-Options, X-Content-Type-Options, HSTS)
  • Rate limiting: 600 requests/minute per IP via express-rate-limit
  • CORS: Enabled globally for cross-origin frontend access

5. Reliability & Concurrency

The Atomic Claim Query

This is the core reliability guarantee of the system. The claim logic in jobRepository.claimJobs() executes inside a single PostgreSQL transaction:

-- Step 1: Lock the queue row (serializes concurrency_limit checks per queue)
SELECT * FROM queues WHERE id = $1 FOR UPDATE

-- Step 2: Count in-flight jobs to enforce concurrency_limit
SELECT count(*)::int FROM jobs
WHERE queue_id = $1 AND status IN ('claimed','running')

-- Step 3: Claim eligible jobs — the critical line
SELECT j.id FROM jobs j
WHERE j.queue_id = $1
  AND j.status IN ('queued', 'scheduled')
  AND j.run_at <= now()
  AND NOT EXISTS (
    SELECT 1 FROM job_dependencies d
    JOIN jobs dep ON dep.id = d.depends_on_job_id
    WHERE d.job_id = j.id AND dep.status <> 'completed'
  )
ORDER BY j.priority DESC, j.run_at ASC, j.created_at ASC
LIMIT $2
FOR UPDATE SKIP LOCKED
Why SKIP LOCKED?
FOR UPDATE SKIP LOCKED means if Worker A is locking a row, Worker B silently skips it — no blocking, no waiting, no deadlocks, and absolutely no double-claiming. The queue-level lock (without SKIP LOCKED) serializes the concurrency_limit check per queue, preventing two workers from both reading "2 of 5 slots used" and each proceeding.

Retry & Backoff Computation

When a job fails, the system evaluates its retry policy:

  • If attempt_count >= max_attempts → permanent failure → Dead Letter Queue
  • Otherwise → compute backoff delay, set run_at = now() + delay, reset status to queued
StrategyFormulaExample (base=10s, mult=2)
Fixeddelay = base_delay10s, 10s, 10s, …
Lineardelay = base_delay × attempt10s, 20s, 30s, …
Exponentialdelay = base_delay × mult^(attempt-1)10s, 20s, 40s, 80s, …

All strategies are capped at max_delay_seconds to prevent unbounded delays.

Worker Lifecycle

  1. Registration: Worker calls POST /workers/register with hostname, PID, queue list, concurrency
  2. Polling: Calls POST /queues/:id/claim every 1 second (configurable)
  3. Execution: For each claimed job: calls /start, runs handler, then /complete or /fail
  4. Heartbeat: Every 5 seconds, sends active job count to the API
  5. Graceful shutdown: On SIGTERM/SIGINT — stops polling, waits for in-flight jobs (15s timeout), marks offline, exits

8. Design Decisions & Trade-offs

PostgreSQL as Queue Engine vs. Redis/RabbitMQ

Decision: Use PostgreSQL's FOR UPDATE SKIP LOCKED instead of a dedicated message broker.

Advantages: Single data store (no state synchronization between two systems), full ACID compliance (crashed workers cannot corrupt state), simpler deployment (one fewer service to provision and monitor).

Disadvantages: Lower raw throughput than an in-memory broker. Postgres handles thousands of claims/second; Redis handles hundreds of thousands. Potential index bloat under sustained high load requires aggressive autovacuum tuning.

When I'd reach for Redis: If the system needed sub-millisecond latency, 100K+ jobs/second, or pub/sub fan-out to thousands of workers.

Decoupled Workers vs. In-Process Execution

Decision: Workers run as separate processes that communicate exclusively via HTTP.

Advantages: True horizontal scaling (50 workers on 50 machines), fault isolation (worker crash doesn't affect API), security in multi-tenant scenarios (project-scoped API key authentication).

Disadvantages: Added network latency for every operation, operational overhead for users who must manage a separate process.

WebSockets vs. Polling

Decision: Persistent WebSocket connection for dashboard updates rather than REST polling.

Advantages: Instant updates (milliseconds vs. seconds), lower database load (one broadcast vs. N polling queries).

Disadvantages: Stateful connections; scaling horizontally would require a Redis pub/sub backplane to sync across API instances.

Vanilla Frontend vs. React/Vue

Decision: No framework — two JS files, no build step.

Rationale: The dashboard is a read-heavy monitoring tool with five views, tables, charts, and modals. A framework would add hundreds of KB of JS and a build pipeline without meaningfully improving the UX for this use case.

jobLifecycle.test.js — 6 tests

  • Creates a job in queued state
  • Idempotency key prevents duplicate creation (409)
  • Full happy path: claim → start → complete (with execution record verification)
  • Job exceeding max_attempts moves to Dead Letter Queue
  • Cancelled job cannot be claimed
  • Paused queue yields no claimable jobs

concurrency.test.js — 2 tests

Critical Concurrency Test
10 workers fire Promise.all claim requests against 20 jobs simultaneously. The test verifies: (a) no job ID appears twice across all results, (b) all claimed IDs belong to the original set, (c) total claimed respects the queue's concurrency_limit. This proves the FOR UPDATE SKIP LOCKED mechanism works correctly under real concurrent load.

Worker deployment strategy: Render's free tier only supports Web Services (Background Workers cost $7/month). The Start Command is set to npm run worker & npm start, launching the worker as a background process alongside the API. Despite sharing a container, the architecture remains strictly decoupled — the worker communicates with the API via HTTP REST calls using the same endpoints and API key authentication it would use from a separate server. Zero code changes would be needed to move the worker to its own service.

End-to-end verification: The live deployment was tested from a smartphone over cellular data — completely disconnected from the development machine. Registration, login, job creation, and job completion all succeeded, confirming zero localhost dependency.
